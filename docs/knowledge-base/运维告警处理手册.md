# 运维告警处理手册

## 一、告警总览

本文档涵盖以下常见告警场景的处理流程：
- **HighCPUUsage** - CPU 使用率过高
- **SlowResponse** - 服务响应时间慢
- **HighMemoryUsage** - 内存使用率过高
- **DatabaseConnectionPoolExhausted** - 数据库连接池耗尽
- **RedisConnectionTimeout** - Redis 连接超时
- **InstanceDown** - 服务实例不可用

---

## 二、CPU 使用率过高 (HighCPUUsage)

### 2.1 告警描述
服务 CPU 使用率持续超过阈值（通常为 80%）

### 2.2 常见原因
- 流量突增或异常流量攻击
- 死循环或无限递归
- 大量并发请求
- 内存不足导致频繁 GC
- 进程异常占用

### 2.3 排查步骤

```bash
# 1. 查看 CPU 使用TOP进程
top -o %CPU

# 2. 查看特定进程线程
top -H -p <pid>

# 3. 查看网络连接（排查攻击）
netstat -anp | grep ESTABLISHED | wc -l

# 4. 查看 Java 线程堆栈
jstack <pid> > thread_dump.txt

# 5. 检查 GC 日志
jstat -gcutil <pid> 1000 10

# 6. 查看最近的异常日志
tail -1000 /var/log/app.log | grep -i error
```

### 2.4 处理方案

| 场景 | 解决方案 |
|-----|---------|
| **流量突增** | 扩容实例、启用熔断、限流 |
| **死循环** | 分析线程堆栈定位问题代码 |
| **GC 频繁** | 调整 JVM 参数、增大堆内存 |
| **异常进程** | 重启服务或 kill 异常进程 |
| **攻击流量** | 开启防火墙、IP 黑名单 |

### 2.5 预防措施
- 设置合理的自动扩缩容策略
- 配置 CPU 使用率告警阈值
- 定期代码审查防止性能问题

---

## 三、服务响应时间慢 (SlowResponse)

### 3.1 告警描述
服务 P99/P95 响应时间超过阈值

### 3.2 常见原因
- 数据库查询慢
- 线程池阻塞
- 外部 API 调用慢
- 锁竞争
- 内存不足导致频繁 GC

### 3.3 排查步骤

```bash
# 1. 查看慢查询日志
grep "slow query" /var/log/app.log

# 2. 检查数据库连接池状态
curl http://localhost:8080/actuator/health

# 3. 查看线程池状态
curl http://localhost:8080/actuator/metrics/tomcat.threads

# 4. 使用 strace 追踪系统调用
strace -p <pid> -c -T

# 5. 分析 SQL 执行计划
EXPLAIN ANALYZE SELECT * FROM xxx WHERE ...

# 6. 查看网络延迟
ping <db_host>
```

### 3.4 处理方案

| 场景 | 解决方案 |
|-----|---------|
| **慢查询** | 添加索引、优化 SQL |
| **线程池满** | 增大线程池配置、优化线程数 |
| **外部调用慢** | 添加超时设置、异步处理 |
| **锁竞争** | 减少锁粒度、使用乐观锁 |
| **GC 影响** | 调整 JVM 参数 |

### 3.5 预防措施
- 配置慢查询日志记录
- 添加数据库索引监控
- 设置 API 超时时间

---

## 四、内存使用率过高 (HighMemoryUsage)

### 4.1 告警描述
服务内存使用率持续超过阈值（通常为 85%）

### 4.2 常见原因
- 内存泄漏
- 缓存过大
- 大对象分配过多
- JVM 堆内存设置过小
- 内存映射文件未释放

### 4.3 排查步骤

```bash
# 1. 查看内存使用
free -h

# 2. 查看进程内存占用
ps aux --sort=-%mem | head -10

# 3. 分析 Java 堆内存
jmap -heap <pid>

# 4. 生成堆转储
jmap -dump:format=b,file=heap_dump.hprof <pid>

# 5. 分析堆转储（使用 MAT 工具）
# 或使用 jhat 进行简单分析
jhat heap_dump.hprof

# 6. 查看缓存统计
curl http://localhost:8080/actuator/metrics/cache.size
```

### 4.4 处理方案

| 场景 | 解决方案 |
|-----|---------|
| **内存泄漏** | 使用 MAT 分析定位泄漏点 |
| **缓存过大** | 设置缓存过期时间、限制缓存大小 |
| **大对象** | 分批处理、使用流式处理 |
| **堆内存不足** | 增大 Xmx 参数 |
| **未释放资源** | 检查资源释放逻辑 |

### 4.5 预防措施
- 定期进行堆内存分析
- 设置缓存大小限制
- 使用弱引用缓存策略

## 五、数据库连接池耗尽 (DatabaseConnectionPoolExhausted)

### 5.1 告警描述
数据库连接池已达到最大连接数

### 5.2 常见原因
- 连接未正确释放
- 慢查询占用连接
- 连接池配置过小
- 数据库服务响应慢

### 5.3 排查步骤

```bash
# 1. 查看数据库连接状态
SELECT * FROM INFORMATION_SCHEMA.PROCESSLIST;

# 2. 查看连接池配置
curl http://localhost:8080/actuator/env | grep -A 10 "hikari"

# 3. 检查慢查询
SHOW FULL PROCESSLIST;

# 4. 查看连接池指标
curl http://localhost:8080/actuator/metrics/hikaricp.connections

# 5. 检查连接泄漏（使用 HikariCP 泄漏检测）
# 在配置中开启 leak-detection-threshold
```

### 5.4 处理方案

| 场景 | 解决方案 |
|-----|---------|
| **连接泄漏** | 检查代码中的 try-with-resources |
| **慢查询** | 优化 SQL、添加索引 |
| **连接池过小** | 增大 maximum-pool-size |
| **数据库慢** | 检查数据库性能、扩容 |

### 5.5 预防措施
- 配置连接池泄漏检测
- 设置连接超时时间
- 监控连接池使用率

## 六、Redis 连接超时 (RedisConnectionTimeout)

### 6.1 告警描述
Redis 连接超时或连接失败

### 6.2 常见原因
- Redis 服务过载
- 网络问题
- 连接池耗尽
- Redis 内存不足
- 主从切换

### 6.3 排查步骤

```bash
# 1. 检查 Redis 状态
redis-cli ping

# 2. 查看 Redis 内存使用
redis-cli INFO memory

# 3. 查看连接数
redis-cli CLIENT LIST | wc -l

# 4. 检查网络连通性
ping <redis_host>
telnet <redis_host> 6379

# 5. 查看 Redis 日志
tail -100 /var/log/redis/redis-server.log

# 6. 查看客户端连接池状态
curl http://localhost:8080/actuator/metrics/redis.pool.*
```

### 6.4 处理方案

| 场景 | 解决方案 |
|-----|---------|
| **Redis 过载** | 扩容 Redis、启用集群 |
| **网络问题** | 检查网络配置、防火墙 |
| **连接池耗尽** | 增大连接池大小 |
| **内存不足** | 清理缓存、增加内存 |
| **主从切换** | 等待切换完成 |

### 6.5 预防措施
- 使用 Redis 集群
- 配置连接池监控
- 设置合理的超时时间

---

## 七、服务实例不可用 (InstanceDown)

### 7.1 告警描述
服务实例离线或健康检查失败

### 7.2 常见原因
- 进程崩溃
- 服务器宕机
- 网络不可达
- 健康检查失败
- OOM 被系统杀死

### 7.3 排查步骤

```bash
# 1. 检查服务器是否可达
ping <server_ip>

# 2. 检查端口是否开放
telnet <server_ip> <port>

# 3. 检查进程状态
ps aux | grep <service_name>

# 4. 查看系统日志（OOM 检查）
dmesg | grep -i oom

# 5. 查看服务日志
tail -200 /var/log/app.log

# 6. 检查 Docker 容器状态（如果是容器部署）
docker ps | grep <container_name>
docker logs <container_name>

# 7. 检查健康检查接口
curl http://<server_ip>:<port>/actuator/health
```

### 7.4 处理方案

| 场景 | 解决方案 |
|-----|---------|
| **进程崩溃** | 重启服务 |
| **服务器宕机** | 联系运维重启服务器 |
| **网络问题** | 检查网络配置 |
| **OOM 杀死** | 分析内存问题、增加内存 |
| **配置错误** | 检查配置文件 |

### 7.5 预防措施
- 配置自动重启策略
- 设置健康检查告警
- 配置服务发现和自动故障转移

## 八、告警处理流程

### 8.1 标准处理流程

```
告警触发 → 确认告警 → 初步诊断 → 定位根因 → 实施修复 → 验证恢复 → 记录总结
    ↓          ↓          ↓           ↓          ↓          ↓          ↓
  监控系统   确认状态   查看指标   分析日志   执行操作   检查状态   归档报告
```

### 8.2 处理优先级

| 优先级 | 告警类型 | 响应时间 |
|-------|---------|---------|
| **P0** | InstanceDown、数据丢失 | 立即 |
| **P1** | HighMemoryUsage、连接池耗尽 | 5分钟内 |
| **P2** | HighCPUUsage、SlowResponse | 15分钟内 |
| **P3** | RedisConnectionTimeout | 30分钟内 |

### 8.3 复盘流程

每次告警处理完成后，需进行复盘：
1. 告警是否准确触发
2. 根因是否找到
3. 修复是否有效
4. 是否需要优化告警阈值
5. 是否需要更新运维文档

## 九、常用命令速查

### 9.1 系统监控

```bash
# CPU 使用
top -o %CPU
mpstat 1 10

# 内存使用
free -h
vmstat 1 5

# 磁盘使用
df -h
iostat -x 1 5

# 网络连接
netstat -anp
ss -tuln

# 进程管理
ps aux --sort=-%mem
kill -9 <pid>
```

### 9.2 Java 应用

```bash
# JVM 信息
jinfo <pid>

# 堆内存
jmap -heap <pid>

# 线程堆栈
jstack <pid>

# GC 统计
jstat -gcutil <pid> 1000 10

# 堆转储
jmap -dump:format=b,file=heap.hprof <pid>
```

### 9.3 日志分析

```bash
# 实时查看日志
tail -f /var/log/app.log

# 搜索错误
grep -i error /var/log/app.log

# 统计错误数量
grep -c "ERROR" /var/log/app.log

# 按时间范围过滤
sed -n '/2024-01-15 14:00:00/,/2024-01-15 15:00:00/p' /var/log/app.log
```

## 附录：告警配置建议

### 告警阈值参考

| 指标 | 警告阈值 | 严重阈值 | 采集间隔 |
|-----|---------|---------|---------|
| CPU 使用率 | > 70% | > 90% | 1分钟 |
| 内存使用率 | > 80% | > 90% | 1分钟 |
| 响应时间 P99 | > 2s | > 5s | 1分钟 |
| 错误率 | > 1% | > 5% | 1分钟 |
| 连接池使用率 | > 80% | > 95% | 30秒 |

### 告警抑制规则

```yaml
# 当服务实例宕机时，抑制该实例的其他告警
inhibit_rules:
  - source_match:
      alertname: InstanceDown
    target_match:
      severity: warning
    equal: ['instance']
```
