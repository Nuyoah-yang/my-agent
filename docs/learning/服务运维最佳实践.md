# 服务运维最佳实践

## 一、概述

本文档总结了微服务架构下的运维最佳实践，涵盖服务部署、监控、故障处理、性能优化等方面。

---

## 二、服务部署规范

### 2.1 容器化部署

**Dockerfile 最佳实践**：

```dockerfile
# 使用多阶段构建
FROM openjdk:21-jdk-slim AS builder
WORKDIR /app
COPY pom.xml .
COPY src ./src
RUN mvn clean package -DskipTests

FROM openjdk:21-jdk-slim
WORKDIR /app
COPY --from=builder /app/target/*.jar app.jar

# 非 root 用户运行
RUN useradd -m appuser && chown -R appuser /app
USER appuser

# JVM 参数优化
ENV JAVA_OPTS="-Xms512m -Xmx1024m -XX:+UseG1GC -XX:MaxGCPauseMillis=200"

EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]
```

### 2.2 健康检查配置

**Spring Boot Actuator 配置**：

```yaml
management:
  endpoints:
    web:
      exposure:
        include: health,info,metrics,prometheus
  endpoint:
    health:
      show-details: always
  health:
    diskspace:
      enabled: true
      threshold: 1073741824  # 1GB
```

**健康检查端点**：
- `GET /actuator/health` - 健康状态
- `GET /actuator/info` - 应用信息
- `GET /actuator/metrics` - 指标信息

### 2.3 日志规范

**日志级别配置**：

```yaml
logging:
  level:
    root: INFO
    com.example.app: DEBUG
    org.springframework.web: INFO
    com.zaxxer.hikari: WARN
  pattern:
    console: "%d{yyyy-MM-dd HH:mm:ss.SSS} [%thread] %-5level %logger{36} - %msg%n"
```

**日志最佳实践**：
- 使用结构化日志（JSON 格式）
- 记录请求 ID 便于追踪

- 定期清理旧日志

## 三、监控体系建设

### 3.1 指标监控

**核心监控指标**：

| 类别 | 指标 | 说明 |
|-----|------|------|
| **服务指标** | QPS、延迟、错误率 | 请求层面 |
| **资源指标** | CPU、内存、磁盘、网络 | 主机层面 |
| **JVM 指标** | 堆内存、GC 次数、线程数 | JVM 层面 |
| **数据库指标** | 连接数、慢查询、锁等待 | 数据库层面 |
| **业务指标** | 订单量、支付成功率 | 业务层面 |

**Prometheus 指标命名规范**：

```
<namespace>_<subsystem>_<metric_name>_<unit>
```

示例：
- `user_service_request_count_total`
- `order_service_response_time_seconds`
- `payment_service_error_rate`

### 3.2 告警策略

**告警分级**：

| 级别 | 颜色 | 含义 | 响应时间 |
|-----|------|------|---------|
| **CRITICAL** | 红色 | 服务不可用 | 立即 |
| **WARNING** | 黄色 | 性能下降或资源紧张 | 15分钟内 |
| **INFO** | 蓝色 | 信息性通知 | 按需 |

**告警规则示例**：

```yaml
groups:
  - name: service_alerts
    rules:
      - alert: ServiceUnavailable
        expr: up == 0
        for: 1m
        labels:
          severity: critical
        annotations:
          summary: "服务不可用"
          description: "服务 {{ $labels.instance }} 已停止运行"

      - alert: HighErrorRate
        expr: rate(error_count_total[5m]) / rate(request_count_total[5m]) > 0.05
        for: 2m
        labels:
          severity: warning
        annotations:
          summary: "错误率过高"
          description: "错误率 {{ $value }} 超过 5%"
```

### 3.3 分布式追踪

**追踪工具选择**：
- **Jaeger** - 开源分布式追踪系统
- **Zipkin** - 轻量级追踪系统
- **SkyWalking** - 国产分布式追踪系统

**追踪采样策略**：
- **固定采样**：采样率固定（如 1%）
- **基于速率采样**：每秒采样固定数量
- **自适应采样**：根据延迟动态调整

## 四、故障处理流程

### 4.1 故障分类

| 类型 | 定义 | 示例 |
|-----|------|-----|
| **P0** | 核心业务完全不可用 | 数据库宕机、主服务崩溃 |
| **P1** | 核心业务部分不可用 | 部分实例故障、性能严重下降 |
| **P2** | 非核心业务不可用 | 辅助功能故障、告警异常 |
| **P3** | 轻微问题或优化建议 | 日志警告、配置优化 |

### 4.2 故障处理步骤

```
发现故障 → 确认范围 → 初步诊断 → 定位根因 → 实施修复 → 验证恢复 → 复盘总结
     ↓          ↓          ↓           ↓          ↓          ↓          ↓
   监控告警   影响评估   收集信息   分析日志   执行操作   检查状态   归档报告
```

### 4.3 故障恢复策略

| 策略 | 适用场景 | 说明 |
|-----|---------|------|
| **快速重启** | 进程崩溃、配置错误 | 立即恢复服务 |
| **流量切换** | 部分实例故障 | 将流量切到健康实例 |
| **回滚版本** | 新版本引入问题 | 回滚到上一个稳定版本 |
| **降级服务** | 资源紧张 | 关闭非核心功能 |
| **扩容实例** | 流量突增 | 增加服务实例数 |

### 4.4 根因分析方法

**5W1H 分析法**：

| 问题 | 说明 |
|-----|------|
| **What** | 发生了什么？ |
| **When** | 什么时候发生的？ |
| **Where** | 在哪个服务/实例发生？ |
| **Why** | 为什么会发生？ |
| **Who** | 谁受影响？ |
| **How** | 如何重现和修复？ |

**鱼骨图分析**：

```
问题
├─ 人员因素
├─ 流程因素
├─ 技术因素
│   ├─ 代码问题
│   ├─ 配置问题
│   ├─ 基础设施问题
│   └─ 依赖问题
└─ 环境因素
```

## 五、性能优化实践

### 5.1 代码优化

**常见性能问题**：

| 问题 | 优化方案 |
|-----|---------|
| **N+1 查询** | 使用 JOIN 查询、批量查询 |
| **重复计算** | 使用缓存、预计算 |
| **大对象处理** | 分批处理、流式处理 |
| **同步阻塞** | 使用异步、并行处理 |
| **锁竞争** | 减少锁粒度、使用乐观锁 |

**优化示例**：

```java
// 优化前：N+1 查询
List<Order> orders = orderRepository.findAll();
for (Order order : orders) {
    User user = userRepository.findById(order.getUserId());  // 每次循环都查询
}

// 优化后：批量查询
List<Order> orders = orderRepository.findAll();
Set<Long> userIds = orders.stream()
    .map(Order::getUserId)
    .collect(Collectors.toSet());
Map<Long, User> userMap = userRepository.findAllById(userIds)
    .stream()
    .collect(Collectors.toMap(User::getId, Function.identity()));
```

### 5.2 数据库优化

**索引优化**：

```sql
-- 创建联合索引
CREATE INDEX idx_order_user_status ON orders(user_id, status);

-- 删除未使用的索引
DROP INDEX idx_unused ON orders;

-- 查看索引使用情况
SELECT * FROM sys.dm_db_index_usage_stats 
WHERE object_id = OBJECT_ID('orders');
```

**查询优化**：

```sql
-- 避免 SELECT *
SELECT id, name, created_at FROM users WHERE status = 1;

-- 使用 LIMIT 限制结果
SELECT * FROM logs ORDER BY created_at DESC LIMIT 100;

-- 合理使用 JOIN
SELECT o.id, o.amount, u.name 
FROM orders o
JOIN users u ON o.user_id = u.id
WHERE o.status = 'completed';
```

### 5.3 缓存策略

**缓存分层**：

| 层级 | 缓存类型 | 适用场景 | 特点 |
|-----|---------|---------|------|
| **L1** | 本地缓存（Caffeine） | 高频访问数据 | 速度快、内存占用 |
| **L2** | 分布式缓存（Redis） | 跨实例共享数据 | 可扩展、网络延迟 |
| **L3** | 数据库 | 持久化数据 | 可靠、速度慢 |

**缓存模式**：

```java
// Cache-Aside 模式
public User getUser(Long id) {
    // 先从缓存获取
    User user = cache.get(id);
    if (user != null) {
        return user;
    }
    // 缓存未命中，从数据库获取
    user = userRepository.findById(id).orElse(null);
    if (user != null) {
        cache.put(id, user, Duration.ofMinutes(30));
    }
    return user;
}
```

## 六、安全运维

### 6.1 访问控制

**API 安全**：

| 措施 | 说明 |
|-----|------|
| **JWT 认证** | 无状态身份认证 |
| **RBAC 授权** | 基于角色的访问控制 |
| **API 限流** | 防止接口被滥用 |
| **输入校验** | 参数校验、防止注入攻击 |

**安全配置示例**：

```yaml
spring:
  security:
    jwt:
      secret: ${JWT_SECRET}
      expiration: 86400000  # 24小时

# 限流配置
rate-limit:
  enabled: true
  requests-per-second: 100
  burst-capacity: 200
```

### 6.2 数据保护

**数据加密**：

| 数据类型 | 加密方式 | 说明 |
|-----|---------|------|
| **密码** | BCrypt/SCrypt | 单向哈希 |
| **敏感数据** | AES-256 | 对称加密 |
| **传输数据** | TLS/HTTPS | 传输加密 |

**脱敏处理**：

```java
// 手机号脱敏
public String maskPhone(String phone) {
    if (phone == null || phone.length() < 11) {
        return phone;
    }
    return phone.replaceAll("(\\d{3})\\d{4}(\\d{4})", "$1****$2");
}

// 邮箱脱敏
public String maskEmail(String email) {
    if (email == null || !email.contains("@")) {
        return email;
    }
    String[] parts = email.split("@");
    if (parts[0].length() <= 2) {
        return email;
    }
    return parts[0].charAt(0) + "***" + parts[0].charAt(parts[0].length() - 1) + "@" + parts[1];
}
```

## 七、灾备与恢复

### 7.1 数据备份

**备份策略**：

| 类型 | 频率 | 保留时间 | 说明 |
|-----|------|---------|------|
| **每日全量备份** | 每天凌晨 | 7天 | 完整数据备份 |
| **增量备份** | 每小时 | 24小时 | 增量数据备份 |
| **实时同步** | 实时 | - | 主从复制 |

**备份验证**：

```bash
# 定期验证备份完整性
pg_restore --list backup.dump
mysqlcheck --databases app_db

# 定期进行恢复测试
pg_restore -d test_db backup.dump
```

## 八、自动化运维

### 8.1 CI/CD 流程

**流水线阶段**：

```
代码提交 → 代码审查 → 构建 → 测试 → 部署 → 验证 → 监控
    ↓          ↓         ↓       ↓       ↓       ↓       ↓
  Git push  PR Review  Maven   JUnit   Docker  Health  Metrics
```

**部署策略**：

| 策略 | 说明 | 适用场景 |
|-----|------|---------|
| **滚动部署** | 逐个替换实例 | 无状态服务 |
| **蓝绿部署** | 切换流量 | 需要回滚保证 |
| **金丝雀部署** | 渐进式发布 | 风险较高的变更 |

### 8.2 自动化运维工具

| 工具 | 用途 | 说明 |
|-----|------|------|
| **Ansible** | 配置管理 | 自动化部署和配置 |
| **Terraform** | 基础设施即代码 | 云资源管理 |
| **Jenkins/GitLab CI** | CI/CD | 自动化流水线 |
| **Prometheus + Alertmanager** | 监控告警 | 指标采集和告警 |
| **Grafana** | 可视化 | 仪表盘展示 |
