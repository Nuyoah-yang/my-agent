## 一、模块概述

`in` 是 WMS 系统的**入库管理模块**，管理货物从"预期到达"到"最终上架存放"的完整流程。它是货物进入仓库的唯一入口，连接 ERP 系统（上游）和库存系统（下游）。

### 1.1 入库整体流程

```
ERP 推送 / 手动创建
        │
        ▼
┌─────────────────────┐
│  ① 到货通知单       │  告知仓库"有货要来了"
│  ArrivalNotice      │  状态: 新建 → 确认 → 收货中 → 完成/关单/取消
└────────┬────────────┘
         │ 一键收货 / PDA 收货
         ▼
┌─────────────────────┐
│  ② 收货单           │  货物实际到达，扫码收货
│  Receipt            │  自动生成上架单 + 创建库存
└────────┬────────────┘
         │
         ▼
┌─────────────────────┐
│  ③ 验收/复查        │  质检环节（采购入库需要人工验收）
│  AcceptDetail       │  验收 → 合格/不合格 → 复查
│  AcceptExamination  │
└────────┬────────────┘
         │ 验收合格 → 修改库存状态为合格
         ▼
┌─────────────────────┐
│  ④ 上架单           │  将货物从收货库位移到存储库位
│  Putaway            │  待上架 → 执行中 → 已上架
└─────────────────────┘
```

### 1.2 子模块一览

| 子模块 | 核心表 | 作用 |
|--------|--------|------|
| 到货通知单 | `wms_arrival_notice` + `_detail` | 入库流程的源头和主控制器 |
| 收货单 | `wms_receipt` + `_detail` | 货物实收记录 |
| 验收管理 | `wms_accept_detail` + `wms_accept_examination` | 质检结果记录 |
| 上架管理 | `wms_putaway` + `_detail` | 货物上架到存储库位 |
| 上架策略 | `wms_putaway_strategy` + `_detail` | 自动推荐最优上架库位 |
| 拒收明细 | `wms_reject_receive` | 收货时拒收的记录 |
| 实物退货单 | `goods_return_order` | 退货出库的管理 |
| 销售退货统计 | （查询 DTO） | 销售退货数据汇总 |

## 二、核心数据流详解

### 2.1 到货通知单（ArrivalNotice）

**接口路径**: `/wms/arrivalNotice`  
**数据表**: `wms_arrival_notice`（主表）+ `wms_arrival_notice_detail`（明细表）

到货通知单是整个入库流程的**枢纽**，几乎所有入库操作都通过它的接口完成。

#### 主表关键字段

| 字段 | 说明 |
|------|------|
| `arrivalNoticeNo` | 到货通知单号（唯一） |
| `erpOrderNo` | ERP 订单号 |
| `orderType` | 订单类型：`CGRK`=采购入库，`XSTHRK`=销售退货入库，`ZHRK`=召回入库 |
| `orderOrigin` | 来源：1=ERP，2=手动创建 |
| `orderStatus` | 状态：1=新建，2=收货中，3=收货完成，4=已关单，5=已取消 |
| `confirmStatus` | 确认状态：1=未确认，2=已确认 |
| `inspectAcceptStatus` | 验收状态：1=未验收，2=已验收，3=部分验收 |

| `ownerCode` / `ownerName` | 货主 |
| `supplierCode` / `supplierName` | 供应商 |

### 2.4 验收状态流转

```
验收状态：1=未验收 → 2=已验收 → 3=部分验收

验收明细状态：
┌──────┐    验收    ┌──────┐   复查申请   ┌──────┐
│已验收 │ (1)  ←──→│复查中 │ (2)          │已取消 │ (0)
└──────┘           └──────┘               └──────┘
                    复查提交
                    合格/不合格
```

### 2.5 订单类型枚举（EOrderType）

| 编码 | 名称 | 是否需要人工验收 |
|------|------|:--:|
| `CGRK` | 采购入库 | 是 |
| `XSTHRK` | 销售退货入库 | 是 |
| `ZHRK` | 召回入库 | 否 |

**关键区别**：采购入库和销售退货入库需要人工验收（质检），召回入库则不需要。这影响收货后是否自动创建验收流程。

## 三、核心业务操作分析

### 3.1 一键收货（oneKeyReceive）

这是入库最核心的操作，**同时完成收货 + 质检 + 上架三件事**：

```
入参：到货通知单ID列表 + 仓库 + 库区 + 库位

流程：
1. 查询到货通知单（含明细）
2. 自动生成收货单 + 收货明细
3. 创建库存记录（库存状态=待检/"合格"）
4. 如果 orderType.needInspectAccept=true：
   → 创建验收记录（待验收状态）
   → 库存状态=待检
5. 如果 orderType.needInspectAccept=false：
   → 自动验收通过
   → 库存状态=合格
6. 自动生成上架单（待上架状态）
7. 更新到货通知单的收货数量、收货状态
```

### 3.2 收货撤销（revokeReceipt / revokeReceiptDetail）

支持**整单撤销**和**明细级撤销**。撤销收货时：
- 撤回库存记录
- 撤回已生成的上架单
- 撤回已生成的验收记录
- 撤回已提交 ERP 的收货结果

### 3.3 验收（inspectAcceptDetail）

质检员对已收货的货物逐项检验，录入：
- 合格数量 / 不合格数量
- 不合格原因
- 抽样数量
- 上传质检检验报告

验收结果会更新：
- 库存状态（合格 → 可销售；不合格 → 待处理）
- 验收明细表
- 到货通知单的验收状态

### 3.4 复查（AcceptExamination）

质检不合格的货物可以进行**复查**：
- 从验收明细中**拆分**出复查记录
- 复查单号关联验收明细
- 复查提交后可修改验收结果

### 3.5 上架策略（PutawayStrategy）

**接口路径**: `/system/putawayStrategy`  
**数据表**: `wms_putaway_strategy` + `wms_putaway_strategy_detail`

上架策略是一组**规则引擎**，决定货物收货后推荐上架到哪个库位。支持：

| 规则 | 说明 |
|------|------|
| 库位排序规则 | 按上架线路顺序升序/降序 |
| 库位堆栈规则 | 先堆满一个库位或均匀分布 |
| 空库推荐规则 | 优先推荐空库位、有同货品的库位或不限制 |
| 匹配规则 | 按货品分类/仓库/库区匹配 |

**策略配置**包含多条明细，每条明细指定：
- 仓库 + 库区范围
- 货品分类
- 排序规则 + 堆栈规则 + 空库推荐规则 + 匹配规则
- 优先级、启用状态

## 四、退货相关

### 4.1 销售退货统计（SalesReturn）

**接口路径**: `/salesReturn`

按销售订单号、客户、货品等维度查询销售退货数据，提供分页和导出。数据来源于到货通知明细表。

### 4.2 实物退货单（GoodsReturnOrder）

**接口路径**: `/goods-return-order`

管理实际退回仓库的货物订单。主要功能：
- 分页查询和导出
- `returnFinish` — 标记退货完成
- `pushToWechat` — 推送企业微信审批

## 五、拒收管理

### 5.1 拒收明细（RejectReceive）

**接口路径**: `/wms/reject`

收货时发现的**不合格货物**，记录拒收明细。提供分页和导出功能。拒收数据关联到到货通知单，影响订单的拒收数量统计。

### 6.4 与 ERP 的交互

入库流程中多处与 ERP 交互：
1. **ERP 推送到货通知**：`erpAdd()` 接收 ERP 推送的到货通知数据
2. **ERP 取消订单**：`erpCancel()` 接收 ERP 取消指令
3. **上报收货状态**：收货/撤销收货后回调 ERP（`ReceiptingDTO`、`ReceiptFinishDTO`）
4. **上报验收结果**：验收完成后同步给 ERP
5. **查询 ERP 客户/供应商**：新增到货通知时查询

## 八、小结

`in` 模块的核心要点：

1. **到货通知单是枢纽**：所有入库操作围绕它展开，它既是数据载体也是流程控制器
2. **一键收货是核心流程**：`oneKeyReceive()` 一行代码触发收货→验收→库存→上架四个动作
3. **订单类型决定验收方式**：采购入库和销退入库需要人工验收，召回入库自动验收
4. **ERP 双向同步**：数据从 ERP 来（到货通知），结果要回传给 ERP（收货、验收结果）
5. **上架策略解耦**：推荐上架库位的逻辑独立为策略模块，支持灵活配置

**下一步**建议阅读 **`io`（库存操作）** 模块，你将看到入库操作如何实际影响库存数据。
